# UI – Common Toast

이 문서는 서비스 전반에서 사용되는 **Toast(피드백 메시지 UI)**의 공통 규칙과 UX 정책을 정의한다.

---

## 1. 역할 정의 (Role)

Toast는 사용자 행동에 대한 **즉각적인 피드백**을 제공하기 위한 UI이다.

### Toast가 적합한 경우

- 성공 / 실패에 대한 간단한 알림
- 비차단(non-blocking) 피드백
- 사용자의 흐름을 끊지 않아야 하는 메시지

### Toast가 부적합한 경우

- 사용자의 확인/결정이 필요한 경우 → Modal 사용
- 중요한 정보 전달, 장문 설명 → 페이지 또는 Modal
- 필수 조치가 필요한 오류 → 화면 내 Error UI

---

## 2. 기본 UX 원칙

- Toast는 **비차단 UI**로 동작한다.
- 사용자의 현재 작업 흐름을 방해하지 않는다.
- 동일한 메시지의 과도한 반복 노출을 방지한다.

---

## 3. Mobile First 배치 정책

### 기본 위치

- 모바일: **화면 하단 중앙**
- 데스크톱: 하단 중앙 또는 우측 하단

> 하단 고정 CTA(Button)와 겹치지 않도록 offset을 둔다.

---

## 4. Variant 설계 (CVA 기준)

Toast는 CVA 기반 variant로 관리한다.

### 4.1 Intent (의미)

- `success`: 작업 성공
- `error`: 작업 실패
- `warning`: 주의가 필요한 상태
- `info`: 일반 정보

### 4.2 Tone

- 기본 톤은 **친절하고 부드러운 표현**을 사용한다.
- error라도 공격적이거나 위협적인 표현은 사용하지 않는다.

---

## 5. 노출 & 해제 정책

### 자동 해제 (Auto Dismiss)

- 기본 노출 시간: **3초**
- error Toast는 4~5초까지 허용 가능

### 수동 해제

- 사용자가 닫을 수 있는 close 버튼 제공 가능
- close 버튼은 선택 사항 (권장)

---

## 6. 메시지 가이드

### 메시지 길이

- 1~2줄 이내를 권장
- 한 문장 또는 짧은 두 문장 구성

### 표현 원칙

- 무엇이 발생했는지 명확히 전달
- 가능하면 다음 행동을 암시

예시:

- ⭕ 저장이 완료되었어요
- ⭕ 네트워크 오류가 발생했어요. 다시 시도해 주세요
- ❌ 알 수 없는 오류가 발생했습니다

---

## 7. 접근성 (Accessibility)

- Toast는 스크린 리더에 인식되어야 한다.
- `aria-live="polite"` 사용을 권장한다.
- 키보드 포커스를 강제로 이동시키지 않는다.

---

## 8. 중복 & 스택 정책

- 동일 메시지는 연속으로 중복 표시하지 않는다.
- 동시에 여러 Toast가 발생할 경우:

  - 최대 3개까지 스택 허용
  - 최신 Toast가 상단에 표시

---

## 9. Button / Action 포함 여부

- Toast는 기본적으로 **Action을 포함하지 않는다**.
- 예외적으로 허용되는 경우:

  - 실행 취소(Undo)

> Action이 2개 이상 필요하면 Modal을 사용한다.

---

## 10. 확장 고려

- Promise 기반 Toast (`loading → success/error`)
- 전역 Toast Store 도입
- 서버 에러 코드별 메시지 매핑

---

### 요약

- Toast는 흐름을 방해하지 않는 피드백 UI
- Mobile First 하단 배치
- CVA 기반 intent 관리
- 친절한 메시지 톤 유지
- Modal과 명확한 역할 분리
